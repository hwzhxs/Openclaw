## Slack: OpenClaw config options to reduce duplicate replies (11:25-11:35 UTC)
- User question: what benefits from setting `channels.slack.thread.initialHistoryLimit: 50` + per-channel `systemPrompt`.
- Conclusion: these improve *context completeness* (more thread history) and make rules less likely to be pushed out of context (channel-level prompt injection), but they do not eliminate concurrency.
- Evaluated Scheme 2: `requireMention: true`.
  - Expected effect: strong reduction of duplicate replies for non-`@all` messages (only mentioned agent triggers); more human-like “named speaker” behavior.
  - Tradeoffs: user must @mention; `@all/@here/@channel` may still trigger multiple agents depending on mention-gating; thread follow-ups without @ may not trigger.
- Evaluated `allowBots: true` + Admin-as-relay/host for serial handoffs.
  - Benefit: can approximate serialized multi-agent responses (especially for `@all`) by having Admin explicitly @mention next agent after prior replies are visible.
  - Major risk: bot-bot triggering can cause infinite loops / echo-chamber without strict gating.
  - Proposed mitigations: explicit handoff marker (only messages with marker can trigger), hard TTL/max-step limit per thread, “only Admin may @mention agents” rule, requireMention-based gating, and enabling `allowBots` only for `#agent-team` to limit blast radius.
